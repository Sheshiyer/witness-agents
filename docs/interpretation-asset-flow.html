Implementation Architecture

Witness Agents Runtime Flow

This document maps the actual code architecture: API and CLI entry points, mode intake gates, Selemene fetch, section orchestration, RAG grounding, inference adapters, asset packaging, audit gates, and NotebookLM integration.

Architecture Claim

The interpreter is a set of bounded components. Context and mode policy gate the run; deterministic engine data becomes a FactLock; orchestration executes atomic tasks; asset generation consumes audited source packs.

01 · Actual Entry Points

There Are Three Runtime Entrances

Standalone API

HTTP handlers validate birth data, rate limits, API keys, tier, promo metadata, and initialize the interpretation pipeline.

Integrated Reading CLI

Multi-person mode runner that loads subject configs, resolves L1-L5 level, validates mode context, acquires solo syntheses, and runs mode passes.

Premium Asset CLI

Local/NotebookLM asset generator that loads deterministic engine JSON and reading markdown, builds source packs, gates drift, and downloads artifacts.

03 · Reading Pipeline

How The Interpreter Actually Runs

Mode Runner Path

  • Parse mode document.
  • Resolve L1-L5 register.
  • Validate mode context.
  • Load subjects and solo syntheses.
  • Execute pass plan.
  • Assemble markdown, metrics, SVG, HTML.

Section Service Path

  • Build FactLock from engine outputs.
  • Create section graph.
  • Run Western, Vedic, Somatic tasks in wave 1.
  • Run synthesis in wave 2.
  • Assemble and repair if needed.
04 · Orchestration Math

Task Execution Is A Dependency Graph

T = {western, vedic, somatic, synthesis}
D(synthesis) = {western, vedic, somatic}
Wave₁ = parallel(T \ synthesis)
Wave₂ = synthesis(Wave₁ outputs)

prompt = FactLock + Grounding + System + User + Reminder

The orchestrator topologically sorts tasks into waves. Each task can request grounding. Retrieved passages are injected after FactLock and before the task system prompt. The FactLock is repeated in both system and user prompt so facts remain the spine.

07 · Asset Factory Architecture

Assets Consume Audited Sources

The asset factory loads deterministic engine JSON and reading markdown. It extracts typed facts, writes a recipient-facing source pack, runs gates, then produces local assets and optionally calls NotebookLM. NotebookLM receives curated markdown, not raw internal engine dumps.

source_pack = dossier + anchors + policy + context + briefs
local_assets = render(source_pack, reading)
notebook_assets = NotebookLM(source_pack) if gate = pass

09 · Data Scope Separation

Same Engine, Different Meaning By Scope

Natal Panchanga

Birth moment imprint. Never called today’s weather.

Current Panchanga

Requires explicit daily mode/current time. May be called current-day timing.

Generated Reading

Texture and synthesis. Never overrides deterministic anchors.

11 · NotebookLM Boundary

NotebookLM Is Downstream, Not The Source Of Truth

Uploads

Every markdown file in the audited source pack becomes a NotebookLM source: dossier, mode policy, answered context, anchors, briefs, boundaries.

Artifacts

Audio, video, slide decks, reports, quiz, flashcards, and mind map are product surfaces generated from the source pack.
