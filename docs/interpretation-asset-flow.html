Witness Agents Architecture

Interpretation → Asset Flow

A visual map of how user intake, mode policy, deterministic Selemene engine data, interpretation outputs, source gates, and NotebookLM assets connect. The goal is to prevent assumptions and keep generated assets grounded in deterministic anchors.

Entry

Question First

Every reading mode requires answered mode context before generation. Missing answers block the run.

Policy

Mode × Register

asset-mode-policy.ts defines solo, married, partner, general synastry, family, and business policies across L1-L5.

Facts

Deterministic Anchors

Engine outputs are fact anchors. Generated prose is texture only, especially for synastry/composite readings.

Gate

Block Drift

Audit/gate blocks nakshatra drift, false somatic absence, natal Panchanga-as-current weather, and missing partner facts.

End-to-End Flow

Where Assumptions Are Blocked

1. Reading Start

  • integratedreading-mode.ts requires --mode-context.
  • validateModeContext() prints mode-specific questions and exits if missing.
  • Mode policy enters prompts as mode_policy and mode_context.

2. Fact Extraction

  • Engine JSON becomes deterministic anchors.
  • Natal Panchanga is separate from current-day Panchanga.
  • Synastry requires partner-level anchors; generated prose is texture only.

3. Source Pack

  • 00b-mode-register-policy.md records mode/register.
  • 00c-answered-mode-context.md records answered questions.
  • NotebookLM receives source pack markdown, not raw internal engine dumps.

4. Audit Gate

  • Blocks nakshatra drift.
  • Blocks false somatic absence.
  • Blocks natal Panchanga being described as current weather.
  • Blocks synastry without partner facts.
