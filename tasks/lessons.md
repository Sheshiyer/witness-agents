# Lessons

- When updating Witness product specs, default to additive evolution framing unless the user explicitly asks for a breaking contract redesign. Separate `current live contract`, `proposed additions`, and `future-state product goals` so documentation does not imply backend churn that is not intended.
- When improving witness observability or reporting, preserve the current promo and entitlement behavior unless the user explicitly asks to change pricing, promo duration, or gating logic.
